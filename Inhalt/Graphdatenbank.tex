\section{Graphbasiertes Datenmodell und Schnittstellendesign}
Der IFC-Standard ist ein stark vernetztes und hierarchisches Datenmodell, bei welchem die Beziehungen selbst oft eine sehr wichtige Rolle einnehmen. Während relationale Datenbanken die Daten in starre Tabellen zwingen und Beziehungen
aufwändig über rechenintensive \texttt{JOIN}-Operatoren modelliert werden müssen, können Graphdatenbanken diese effizient darstellen \cite{alkhatib2021graph}. Somit können Datenpunkte mit deren zugehörigen Beziehungen gezielt abgefragt werden. Zudem kann aufgrund
der graphbasierten Struktur der IFC-Standard realitätsnah und intuitiv dargestellt werden. 


\subsection{Graphbasierte Domänenmodellierung für IFC-Daten}
Das Ziel der Domänenmodellierung ist es, mithilfe der Graphdatenbank eine möglichst realitätsnahe und intuitive Abbildung der bereits bestehenden IFC-Dateien – einen sogenannten \textit{digitalen Zwilling} – zu erschaffen. 
IFC-Dateien nutzen das \gls{STEP}-Dateiformat, um ihren Inhalt zu beschreiben. 
\begin{lstlisting}[language=IFC, caption={Beispielhafter IFC-Datenauszug im STEP-Dateiformat (ISO 10303-21).}, label={lst:ifc_step_example}]
ISO-10303-21;
HEADER;
FILE_DESCRIPTION(('ViewDefinition [CoordinationView_V2.0]'),'2;1');
FILE_NAME('Bautagebuch_Modell.ifc','2026-06-02T22:15:00',('Architekt Max'),('Ingenieurbuero X'),'IFC-Exporter v1.0','CAD-BIM-Software 2026',_ );
FILE_SCHEMA(('IFC4'));
ENDSEC;
DATA;
10= IFCPROJECT('0Yn$b7pX187R3W84z_uXy1',15,'Projekt Grossbaustelle IBN','Beschreibung des Grossprojekts',$,$,$,(#40),#30);
15= IFCOWNERHISTORY(#20,#25,$,.ADDED.,$,$,$,1717366500);
20= IFCPERSONANDORGANIZATION(#21,#22,$);
21= IFCPERSON('ID_01','Mustermann','Max',$,$,$,$,$);
22= IFCORGANIZATION('ORG_01','Bauunternehmung GmbH',$,$,$);
25= IFCAPPLICATION(#22,'v1.0','BIM-Software','App_ID');
\end{lstlisting}
Dieses Schema lässt sich bereits als eine Art Graph verstehen. Die Objekte selbst können als Knoten mit ihren Attributen als Daten und die Verweise der Objekte (z.B. \#10 verweist auf \#15) als Kanten, also deren Beziehungen, interpretiert werden.

Durch ein graphbasiertes Domänenmodell lässt sich der IFC-Standard einfach, nahezu verlustfrei \cite{giehl} und realitätsnah abbilden \cite{ismail}. \subsection{GraphQL als Vermittlungsinstanz}
Für eine effiziente und IFC-konforme Kommunikation wird als Vermittlungsschicht GraphQL zwischen den Clients und der Graphdatenbank implementiert. Diese Middleware hilft bei der Lösung zentraler Herausforderungen, indem sie Lösungen zu Typsicherheit und detailliertem
Datenabruf bereitstellt. 
Während Knoten in Graphdatenbanken beliebig definiert werden können und somit eine konsequente Einhaltung des IFC-Standards erschwert, können mittels GraphQL-Schemata definiert werden, um genau dies zu verhindern \cite{wolf_20206_graphql}.

Folgend ist exemplarisch ein Schema für die IFC-Klasse \textit{IfcObject} und deren Superklassen dargestellt:
\begin{lstlisting}[caption={GraphQL-Schema für IFC-Kernausschnitte}, label={lst:graphql_ifc}]

interface IfcObjectDefinition implements IfcRoot {
  globalId: ID!
  name: String
  description: String
}

interface IfcObject implements IfcRoot & IfcObjectDefinition {
  globalId: ID!
  name: String
  description: String
  objectType: String
}
interface IfcProduct implements IfcRoot & IfcObjectDefinition & IfcObject {
  globalId: ID!
  name: String
  description: String
  objectType: String
  objectPlacement: String
}
\end{lstlisting}
Durch das Definieren dieser Schemata kann mit der Nutzung von GraphQL eine strikte Typvalidierung durchgesetzt werden. Dies forciert IFC-konforme Datenmodelle innerhalb der Graphdatenbank, indem GraphQL serverseitig 
eingehende Anfragen gegen die Schemata validiert \citep{wolf:2025:ifc2graphql}. Gleichzeitig können Änderungen der Graphdatenbank mittels Mutationen gesteuert werden, sodass diese als Validierungsinstanz fungieren, um die IFC-Konformität zu gewährleisten.
Da jede Mutation einem strikt typisierten Schema entsprechen muss, verhindert dies ein unkontrolliertes Einbringen nicht-standardisierter Objekte. Wie in der Darstellung \ref{lst:graphql_ifc} zu sehen ist, verhält sich GraphQL anders als
populäre Programmiersprachen: Attribute werden nicht impliziert von vererbenden Klassen übernommen, sondern müssen erneut deklariert werden. So muss bei der IFC-Klasse \textit{IfcObject} die Attribute \textit{globalId, name} und \textit{description} erneut
deklariert werden, obwohl diese bereits in der vererbenden Klasse \textit{IfcRoot} deklariert wurden.
Ein weiterer Vorteil liegt in der Optimierung von Datenabfragen. Während klassische REST-APIs zu Over- oder Underfetching neigen, eliminiert GraphQL diese Phänomene vollständig. Die Clients definieren durch ihre Queries selbstständig, welche Daten
sie benötigen. Diese Queries werden von Resolvern (siehe \ref{sec:graphql}) verarbeitet, indem sie die Anfragen in die Abfragesprache Cypher übersetzen, um effizient die Daten aus der Graphdatenbank abzurufen.
\subsection{GraphQL als einheitliche API}
In einer Add-on-fähigen Softwarearchitektur, in der ggf. unterschiedliche Entwickler arbeiten, ist eine flexible API von großem Vorteil. Während REST-APIs auf starre vordefinierte Endpunkte setzen, basiert GraphQL auf clientgesteuerten Abfragen. Somit können
Entwickler benötigte Schnittstellen flexibel und eigenständig über die bestehenden Schemata abbilden. Zudem können Clients mittels der sogenannten \textit{Schema Introspection} die vorhandenen Schemata abfragen. Durch diese Transparenz entfallen sonst notwendige
Dokumentationen, wodurch die Einstiegshürden für neue Entwickler drastisch gesenkt werden und das Erstellen neuer Add-Ons auf Basis des Domänenmodells vereinfacht wird.

Neben diesen Vorteilen bietet das Verwenden von GraphQL als Middleware eine wesentlich zugänglichere Abstraktionsebene als gängige Abfragesprachen für Graphdatenbanken wie Cypher oder SPARQL.
