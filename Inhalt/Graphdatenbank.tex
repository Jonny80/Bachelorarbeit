\section{Graphbasiertes Datenmodell und Schnittstellendesign}
Der IFC-Standard ist ein stark vernetztes und hierarchisches Datenmodell, bei welchem die Beziehungen selbst oft eine sehr wichtige Rolle einnehmen. Während relationale Datenbanken die Daten in starre Tabellen zwingen und Beziehungen
aufwändig über rechenintensive \texttt{JOIN}-Operatoren modelliert werden müssen, können Graphdatenbanken diese effizient darstellen \cite{alkhatib2021graph}. Somit können Datenpunkte mit deren zugehörigen Beziehungen gezielt abgefragt werden. Zudem kann aufgrund
der graphbasierten Struktur der IFC-Standard realitätsnah und intuitiv dargestellt werden.


\subsection{Graphbasierte Domänenmodellierung für IFC-Daten}
Das Ziel der Domänenmodellierung ist es, mithilfe der Graphdatenbank eine möglichst realitätsnahe und intuitive Abbildung der bereits bestehenden IFC-Dateien – einen sogenannten \textit{digitalen Zwilling} – zu erschaffen.
IFC-Dateien nutzen das \gls{STEP}-Dateiformat, um ihren Inhalt zu beschreiben.
\begin{lstlisting}[language=IFC, caption={Beispielhafter IFC-Datenauszug im STEP-Dateiformat (ISO 10303-21).}, label={lst:ifc_step_example}]
ISO-10303-21;
HEADER;
FILE_DESCRIPTION(('ViewDefinition [CoordinationView_V2.0]'),'2;1');
FILE_NAME('Bautagebuch_Modell.ifc','2026-06-02T22:15:00',('Architekt Max'),('Ingenieurbuero X'),'IFC-Exporter v1.0','CAD-BIM-Software 2026',_ );
FILE_SCHEMA(('IFC4'));
ENDSEC;
DATA;
10= IFCPROJECT('0Yn$b7pX187R3W84z_uXy1',15,'Projekt Grossbaustelle IBN','Beschreibung des Grossprojekts',$,$,$,(#40),#30);
15= IFCOWNERHISTORY(#20,#25,$,.ADDED.,$,$,$,1717366500);
20= IFCPERSONANDORGANIZATION(#21,#22,$);
21= IFCPERSON('ID_01','Mustermann','Max',$,$,$,$,$);
22= IFCORGANIZATION('ORG_01','Bauunternehmung GmbH',$,$,$);
25= IFCAPPLICATION(#22,'v1.0','BIM-Software','App_ID');
\end{lstlisting}
Dieses Schema lässt sich bereits als eine Art Graph verstehen. Die Objekte selbst können als Knoten mit ihren Attributen als Daten und die Verweise der Objekte (z.B. \#10 verweist auf \#15) als Kanten, also deren Beziehungen, interpretiert werden.
Durch ein graphbasiertes Domänenmodell lässt sich der IFC-Standard einfach, nahezu verlustfrei \cite{giehl} und realitätsnah abbilden \cite{ismail}. Um dies zu erreichen, werden konkrete IFC-Klassen wie \textit{IfcBuildingStorey} oder \textit{IfcSpace} als Knoten dargestellt und Beziehungen wie \textit{IfcRelAggregates} als Kanten.
Die Attribute dieser Klassen werden in den Objekten analog zum IFC-Standard als Eigenschaften (\textit{Properties}) gespeichert.
Dies wird beispielhaft in der Web-Oberfläche der genutzten Graphdatenbank Neo4j veranschaulicht:

ihre äquivalenten IFC-Klassen oder -Beziehungen zu kennzeichnen. Mit dieser Methode lässt sich ein objektorientiertes Domänenmodell erstellen, ähnlich der objektorientierten Softwarearchitektur \cite{giehl}.

\subsection{GraphQL als Vermittlungsinstanz}
Für eine effiziente und IFC-konforme Kommunikation wird als Vermittlungsschicht GraphQL zwischen den Clients und der Graphdatenbank implementiert. Diese Middleware hilft bei der Lösung zentraler Herausforderungen, indem sie Lösungen zu Typsicherheit und detailliertem
Datenabruf bereitstellt.
Während Knoten in Graphdatenbanken beliebig definiert werden können und somit eine konsequente Einhaltung des IFC-Standards erschwert, können mittels GraphQL-Schemata diese definiert werden, um genau dies zu verhindern. \cite{wolf_20206_graphql}.
Folgend ist exemplarisch ein Schema für die IFC-Klasse \textit{IfcObject} und deren Superklassen dargestellt:
\begin{lstlisting}[caption={GraphQL-Schema für IFC-Kernausschnitte}, label={lst:graphql_ifc}]

interface IfcObjectDefinition implements IfcRoot {
  globalId: ID!
  name: String
  description: String
}

interface IfcObject implements IfcRoot & IfcObjectDefinition {
  globalId: ID!
  name: String
  description: String
  objectType: String
}
interface IfcProduct implements IfcRoot & IfcObjectDefinition & IfcObject {
  globalId: ID!
  name: String
  description: String
  objectType: String
  objectPlacement: String
}
\end{lstlisting}
Durch das Definieren dieser Schemata kann mit der Nutzung von GraphQL eine strikte Typvalidierung durchgesetzt werden. Dies forciert IFC-konforme Datenmodelle innerhalb der Graphdatenbank, indem GraphQL serverseitig
eingehende Anfragen gegen die Schemata validiert \cite{wolf:2025:ifc2graphql}. Gleichzeitig können Änderungen der Graphdatenbank mittels Mutationen gesteuert werden, sodass diese als Validierungsinstanz fungieren, um die IFC-Konformität zu gewährleisten.
Da jede Mutation einem strikt typisierten Schema entsprechen muss, verhindert dies ein unkontrolliertes Einbringen nicht-standardisierter Objekte. Wie in der Darstellung \ref{lst:graphql_ifc} zu sehen ist, verhält sich GraphQL anders als
populäre Programmiersprachen: Attribute werden nicht implizit von vererbenden Klassen übernommen, sondern müssen erneut deklariert werden. So ist es notwendig, bei der IFC-Klasse \textit{IfcObject} die Attribute \textit{globalId, name} und \textit{description} erneut
zu deklarieren, obwohl diese bereits in der vererbenden Klasse \textit{IfcRoot} enthalten sind.
Ein weiterer Vorteil liegt in der Optimierung von Datenabfragen. Während klassische REST-APIs zu Over- oder Underfetching neigen, eliminiert GraphQL diese Phänomene vollständig. Die Clients definieren durch ihre Queries selbstständig, welche Daten
sie benötigen. Diese Queries werden von Resolvern (siehe \ref{sec:graphql}) verarbeitet, indem sie die Anfragen in die Abfragesprache Cypher übersetzen, um effizient die Daten aus der Graphdatenbank abzurufen.
\subsection{GraphQL als einheitliche API}
In einer Add-on-fähigen Softwarearchitektur, in der ggf. unterschiedliche Entwickler arbeiten, ist eine flexible API von großem Vorteil. Während REST-APIs auf starre vordefinierte Endpunkte setzen, basiert GraphQL auf clientgesteuerten Abfragen. Somit können
Entwickler benötigte Schnittstellen flexibel und eigenständig über die bestehenden Schemata abbilden. Zudem können Clients mittels der sogenannten \textit{Schema Introspection} die vorhandenen Schemata abfragen. Durch diese Transparenz entfallen sonst notwendige
Dokumentationen, wodurch die Einstiegshürden für neue Entwickler drastisch gesenkt werden und das Erstellen neuer Add-Ons auf Basis des Domänenmodells vereinfacht wird.

Neben diesen Vorteilen bietet das Verwenden von GraphQL als Middleware eine wesentlich zugänglichere Abstraktionsebene als gängige Abfragesprachen für Graphdatenbanken wie Cypher oder SPARQL.
\subsection{Implementierung am Beispiel der Klasse \textit{IfcSensor}}
Eine prototypische Umsetzung wird folgend anhand eines Beispiels der Mess,-Steuer- und Regelungstechnik (MSR) bezogenen IFC-Klasse \textit{IfcSensor} verdeutlicht. Die verwendete Version des IFC-Standards ist 4.3.
\subsubsection{Übertragung der IFC-Klassen in GraphQL-Schemata}
Die Definition der GraphQL-Schemata orientiert sich unmittelbar an den IFC-Spezifikationen, welche von \textit{buildingSMART} bereitgestellt werden. Diese gibt sowohl die Attribute und möglichen Beziehungen
als auch die Vererbungshierarchie der IFC-Klassen vor.
Folgend wird die Vererbungshierarchie der Klasse \textit{IfcSensor} aus der offiziellen Online-Dokumentation auf der Homepage von buildingSMART dargestellt:
\begin{figure}[H]
  \centering

  \caption{Hierarchie der IFC-Klasse IfcSensor \cite{IfcSensor}}
  \label{fig:ifc_sensor_hierarchy}

Wie bereits in Listing \ref{lst:graphql_ifc} ersichtlich, kann diese Vererbungshierarchie in den GraphQL-Schemata abgebildet werden.
Für eine pragmatische Umsetzung wurde jedoch auf die Definition nicht essentieller Beziehungen in den GraphQL-Schemata verzichtet. Ein Beispiel hierbei liefert die abstrakte Klasse
\textit{IfcObjectDefinition} \cite{IfcObjectDefinition}, welche sieben optionale Beziehungen definiert. Die Klasse \textit{IfcSensor} erhöht sich somit von acht auf 37 Definitionen auf \cite{IfcSensor}.
Dies wird in dem folgenden Schema ersichtlich, welches die Implementierung der IFC-Klassen \textit{IfcSensor} verdeutlicht:
\begin{lstlisting}[caption={GraphQL-Schema für die Klasse IfcSensor}, label={lst:graphql_ifcSensor}]
type IfcSensor implements IfcRoot & IfcObjectDefinition &
IfcObject & IfcProduct & IfcElement & IfcDistributionElement
& IfcDistributionControlElement @node {
  globalId: ID!
  name: String
  description: String
  objectType: String
  objectPlacement: String
  tag: String
  predefinedType: String
  ContainedInStructure: [IfcSpatialStructureElement!]!
    @relationship(
      type: "CONTAINED_IN_SPATIAL_STRUCTURE"
      direction: IN
      properties: "IfcRelContainedInSpatialStructure"
    )
}
\end{lstlisting}
Nicht optionale Attribute wie \textit{globalId} werden mit dem Suffix \textit{!} versehen.
Als essenziell werden in diesem Kontext Beziehungen eingestuft, die für die strukturelle Integrität innerhalb der Graphdatenbank erforderlich sind.
Als Beispiel hier dient die Beziehung \textit{ContainedInStructure}. Während die übrigen IFC-Relationen entfallen, verhindert diese Kante aus Sicht der Graphenstruktur die Entstehung isolierter Knoten.
Die strukturelle Notwendigkeit dieser Beziehung wird in Abbildung \ref{fig:graph_isolation} visuell verdeutlicht.
\begin{figure}[H]
  \centering
  \includegraphics[width=0.5\textwidth]{Essential_rel.png}
  \caption{Vergleich der Graphenstruktur: Ohne die Relation \textit{IfcSpatialStructureElement} existiert der Sensor als isolierter Knoten ohne topologische Anbindung.}
  \label{fig:graph_isolation}
\end{figure}
Die Typdefinition für Beziehungen, hier \textit{ContainedInStructure}, erfolgt im GraphQL-Schema mittels der \textit{@relationship} Direktive. Hiermit können Beziehungen definiert beschrieben werden, um sie
in unterschiedlichen Schema-Definitionen wiederzuverwenden. Der folgende Codeausschnitt verdeutlicht das Beispiel der Typdefinition der Beziehung \textit{ContainedInStructure}, wobei \textit{IfcSpatialStructureElement} eine
Superklasse der IFC-Klasse \textit{IfcSpace} ist:
\begin{lstlisting}[caption={GraphQL-Schema für die Beziehung ContainedInStructure}, label={lst:graphql_ContainedInStructure}]
ContainedInStructure: [IfcSpatialStructureElement!]!
  @relationship(
    type: "CONTAINED_IN_SPATIAL_STRUCTURE",
    direction: IN,
    properties: "IfcRelContainedInSpatialStructure"
  )
\end{lstlisting}
\subsubsection{Mutationen und Resolver} \label{sec:Mutations_Resolvers}
Wie bereits in Kapitel \ref{sec:graphql} beschrieben, nutzt GraphQL sogenannte Resolver für die Datenabfrage und Mutationen für die Datenmanipulation. Wie in Abbildung~\ref{lst:graphql_ifcSensor} erkennbar ist, wird
für den Typ \textit{IfcSensor} die \textit{@node} Direktive verwendet. Zum einen signalisiert dies, dass es sich um einen Knoten in der Graphdatenbank handelt, aber zum anderen analysiert
die native GraphQL-Bibliothek von Neo4j, das Schema und generiert automatisiert Mutationen für Lese-Schreib und Aktualisierungsoperationen, sowie Resolver mit standardisierten Filterungs- und Sortierungsoperationen \cite{neo4j_graphql_node}.
Obwohl sich diese Automation nur auf einfache CRUD-Operationen beschränkt, verhilft dies trotzdem zu einem erheblichen Zeitersparnis. Komplexere Abfragen müssen hingegen manuell definiert werden. Dafür stehen
zwei Methoden zur Verfügung:
\begin{itemize}
  \item Bei der \textbf{GraphQL basierten Methode} übersetzt die GraphQL-Bibliothek die GraphQL-Abfragesprache in native Cypher-Abfragen, um mit der Graphdatenbank zu kommunizieren.  Das folgende Beispiel veranschaulicht diesen Prozess:
        \begin{lstlisting}[caption={Clientseitige Abfrage für die Klasse \textit{IfcSensor}}, label={lst:graphql_clientQuery}]
query GetSensorsInErdgeschoss {
  ifcSensors(where: {
    ContainedInStructure: {
      name: "Erdgeschoss"
    }
  }) {
    globalId
    name
    predefinedType
  }
}
\end{lstlisting}
  \item Mit der \textbf{Cypher basierten Methode} können Abfragen direkt in Cypher formuliert werden, wie dieses Beispiel veranschaulicht:
        \begin{lstlisting}[caption={Serverseitige Abfrage für die Klasse \textit{IfcSensor}}, label={lst:graphql_serverQuery}]
type Query {
  sensorsInStructure(structureName: String!): [IfcSensor!]!
    @cypher(
      statement: """
      MATCH (s:IfcSensor)-[:CONTAINED_IN_SPATIAL_STRUCTURE]->(f:IfcSpatialStructureElement)
      WHERE f.name = $structureName
      RETURN s
      """,
      columnName: "s"
    )
}
\end{lstlisting}
\end{itemize}
Die Entscheidung, wann welche Abfragemethode sinnvoll ist, hängt stark von dem Anwendungsfall ab. Wie bereits beschrieben nutzt auch die GraphQL-Bibliothek Cypher, um mit der Graphdatenbank zu kommunizieren. Dabei werden auch automatisiert Performanceprobleme, wie das Lösen des N + 1 Problems (vgl. \cite{neo4j_graphql_node}) gelöst. Das \textsl{N + 1 Problem} beschreibt die Problematik, bei der für eine Liste von \texttt{N} Elementen, insgesamt \texttt{N + 1} Abfragen an die Datenbank geschickt werden, anstatt alle Abfragen in einer einzigen zu kapseln. \newline
Dennoch gibt es Einschränkungen bei der Nutzung von reinen GraphQL-Abfragen. Bei komplexen Graph-Traversierungen, bei denen die genaue Tiefe des Graphs nicht im Voraus bekannt ist, stoßen reine GraphQL-Abfragen an ihre Grenzen. Hier zeigen sich die Vorteile von Cypher, da es semantisch in der Lage ist, die Pfade innerhalb der Graphdatenbank zu analysieren und die Tiefe rekursiv zu durchlaufen \cite{neo4j_cypher_variable_length}.
Im Rahmen von MSR und TGA zeigt das folgende Beispiel die Problematik. Es soll der Controller (\textit{IfcController}), welcher einen Aktor (\textit{IfcActuator}) steuert, ermittelt werden. Hierfür muss der Pfad über mehrere Klassen abgelaufen werden. Die Anzahl der dazwischenliegenden Klassen und der daraus resultierenden Relationen ist jedoch variabel, da die genaue Struktur der Ports und Bus-Kopplungen bei komplexen Anlagen im Vorfeld nicht einheitlich ist. Cypher bietet hierfür eine semantische Lösung:
\begin{lstlisting}[caption={Serverseitige Abfrage für die Klasse \textit{IfcSensor}}, label={lst:graphql_recurive_query}]
MATCH path = (actuator:IfcActuator {name: "Stellantrieb_Heizkreis_V01"})
  <-[:`:IfcRelNests`|`:IfcRelConnectsPorts`*1..15]- (controller:IfcController)
RETURN controller.name AS ControllerName
\end{lstlisting}
Der entscheidende Parameter hierbei ist \textit{*1..15}. Hiermit kann eine variable Pfadlänge angegeben werden, welcher die Graphdatenbank anweist, den Pfad bis zu einer Maximallänge von 15 rekursiv zu durchsuchen.
Eine solche Anweisung lässt sich nicht direkt in reinem GraphQL abbilden, da hier die Pfadtiefe und die zugrunde liegende Struktur explizit angegeben werden müssten.
Da jedoch eine solche Komplexität bei einfachen CRUD-Operationen nicht notwendig ist, bietet GraphQL bei solchen Operationen eine bessere Entwicklererfahrung (Developer-Experience). Dazu steuert die Typsicherheit, sowie die automatisch generierten Mutationen und Resolver bei.


\subsection{Beispiel für MSR und TGA Klassen im IFC Standard}
Bei Großbaustellen steigt die Komplexität von Mess-und Regelungstechnik (MSR) sowie der technischen Gebäudeautomatisierung (TGA) anhand der Anzahl und der Verbindungen der einzelnen Komponenten stark.
Die Struktur der MSR-und TGA-spezifischen Klassen ist dabei durch die Beziehungen untereinander, aber auch durch die Beziehungen mit physischen Räumen wie \textit{IfcSpace} oder organisatorischen Klassen wie \textit{IfcTask} stark vernetzt. Dies soll das Beispiel eines kleinen Regelkreises verdeutlichen:

\subsubsection{Künstliche Intelligenz als Hilfsmittel}
Mit über 1000 unterschiedlichen Klassen in Version 4.3 verfügt die IFC-Spezifikation über einen enormen Umfang. Trotz der Hilfe automatisierter Resolver und Mutationen seitens GraphQL stellt das
manuelle Erstellen der GraphQL-Schemata daher eine große Herausforderung dar. Um diesen Anforderungen entgegenzukommen, wurde im Rahmen dieser Arbeit Künstliche Intelligenz (KI) in Form von
Large Language Models (LLM) zur Generierung der GraphQL-Schemata und spezifischer Resolver und Mutationen genutzt. Als Werkzeug wurde hierbei das Modell \textit{Gemini 3.5 (Medium)} von Google verwendet.
Wichtig zu erwähnen ist, dass beim Einsatz der KI oft eine spezifische Anweisung mit direktem Verweis auf die Dokumentation notwendig war, um das Verwenden der aktuellen IFC-Version zu forcieren und
unerwünschte Optimierungen zu vermeiden. Ohne diese gezielten Anweisungen neigte die KI dazu, veraltete IFC-Versionen zu beschreiben, Attribute auszulassen oder Spezifikationen abweichend von den offiziellen
Vorgaben zu definieren. Daher war eine manuelle Überprüfung der generierten Schemata unumgänglich, um die Korrektheit der Schemata zu gewährleisten. \newline
Trotz dieser Einschränkungen bot die KI eine enorme Unterstützung, womit der Entwicklungsaufwand erheblich reduziert werden konnte.
Zudem muss einschränkend festgehalten werden, dass im Rahmen dieser Arbeit ausschließlich das genannte Modell genutzt wurde. Ein direkter Vergleich mit anderen Modellen fand daher nicht statt,
weshalb nicht ausgeschlossen werden kann, dass alternative Modelle für diesen spezifischen Anwendungsfall eine höhere Präzision oder eine geringere Fehlerquote aufweisen.
