\section{Grundlagen}
\subsection{IFC-Standard}
Der IFC-Standard (Industry Foundation Classes) ist eine offene, internationale Spezifikation zur Beschreibung von Bauelementen und deren Beziehungen
zueinander. Er wird seit 1994 von der Organisation buildingSMART im \gls{Express}-Schema entwickelt, um die Digitalisierung auf Baustellen zu
vereinheitlichen. Er wird von den meisten \ac{BIM}-Softwarelösungen verwendet, die ihre Modellierungen als \texttt{.ifc}-Dateien exportieren können.
Der IFC-Standard ist stark hierarchisch aufgebaut und besteht aus Objekten, Attributen und Beziehungen. So stammen Objekte, die physische Gegenstücke abbilden,
von der Basisklasse \textit{IfcRoot} ab und erben über spezialisiertere Zwischenklassen weitere spezifische Attribute.
Abbildung \ref{fig:wall_hier} zeigt die Hierarchie der Klasse \textit{IfcWall} auf:

    \label{fig:wall_hier}
\end{figure}

Die Objekte werden durch Beziehungen miteinander verknüpft. Beispielsweise ist ein Gebäude (\textit{IfcBuilding}) über die Relation \textit{IfcRelAggregates} mit einem Bauplatz (\textit{IfcSite}) verbunden. Dies ist schematisch in Abbildung \ref{fig:loc_structure} dargestellt:

\subsection{Graphdatenbanken}
Graphdatenbanken organisieren ihre Daten in Form eines Graphen. Dies bedeutet, dass Objekte als Knoten repräsentiert und über Kanten miteinander verbunden werden. Der wesentliche Vorteil dieser Struktur ist die native Darstellung von Beziehungen zwischen Objekten.
Dies wird insbesondere bei Datenmengen wichtig, bei denen die Beziehungen untereinander einen elementaren Bestandteil der Datenerhebung darstellen. Bei den gängigsten Graphdatenbanken handelt es sich um sogenannte Labeled Property Graphs. Hierbei können nicht nur
Knoten, sondern auch Kanten mit Labels und Eigenschaften versehen werden.
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.5\textwidth]{Graph_Database.png}
    \caption{Labeled Property Graph}
    \label{fig:property_graph}
\end{figure}
Dieser Ansatz bietet eine hohe Flexibilität gegenüber relationalen Datenbanken, da hier die Datenobjekte logisch miteinander verknüpft und aufwendige JOIN-Operationen vermieden
werden können \cite{seidel2020graph}. Graphdatenbanken erfreuen sich vor allem großer Beliebtheit bei der Analyse stark vernetzter Daten, beispielsweise in sozialen Netzwerken, bei der Betrugserkennung oder im Lieferkettenmanagement \cite{google_graphdb}, da hierbei die Analyse der Verbindungen im Vordergrund
steht.
\subsubsection{Neo4j}
Neo4j ist eine \gls{acid}-konforme Graphdatenbank auf der Grundlage von Labeled Property Graphs. Sie wurde erstmalig 2007 von Emil Eifrem veröffentlicht und avancierte zu einer der weltweit führenden Graphdatenbanken. Ein zentrales Merkmal von Neo4j ist ihre
Abfragesprache Cypher. Sie ist eine SQL-ähnliche, deklarative Sprache. Das bedeutet, dass die Frage im Fokus steht, wonach gesucht wird. Sie folgt dabei einem einfachen, strukturellen Aufbau:
\begin{itemize}
    \item \textbf{Knoten} werden in Klammern dargestellt: \texttt{(Node)};
    \item \textbf{Beziehungen} werden als Pfeile zwischen den Knoten visualisiert: \texttt{-[:FOLGT]->};
    \item \textbf{Eigenschaften} werden als Schlüssel-Wert-Paare in geschweiften Klammern definiert: \texttt{\{name: 'Max'\}}.
\end{itemize}
Das folgende Listing zeigt beispielhaft eine Abfrage von offenen Mängeln an einer technischen Komponente:
\begin{lstlisting}[language=cypher, caption={Abfrage von offenen Mängeln an einer technischen Komponente}]
MATCH (c:Component {id: 'Pumpe-01'})-[:HAS_ISSUE]->(i:Issue)
WHERE i.status <> 'CLOSED'
RETURN c, i
\end{lstlisting}


\subsubsection{GraphQL} \label{sec:graphql}
Als Alternative zu \gls{Rest-API}s wurde GraphQL 2015 von Meta entwickelt. Der Vorteil liegt in der Flexibilität und der erhöhten Effizienz der Abfragesprache. Im Gegensatz zu REST kann mit GraphQL die Anfrage strukturiert werden, um nur die benötigten
Daten zu erhalten und so das sogenannte Over- oder Underfetching stark zu reduzieren. Zusammengefasst kann der Entwickler genau definieren, welche Daten er abfragen möchte. Um dies zu ermöglichen, nutzt GraphQL vier Schlüsselkonzepte:
\begin{itemize}
    \item Schemata
    \item Resolver
    \item Mutationen
    \item Abfragen
\end{itemize}
Das Herzstück von GraphQL sind die \texttt{Schemata}. Sie definieren die Struktur der Daten, welche abgefragt werden können, und werden in der stark typisierten Schemadefinitionssprache (SDL) geschrieben.

Während Schemata die Struktur der Daten definieren, bestimmen \texttt{Resolver}, wie und woher diese Daten bezogen werden. Jedes Feld in einem Schema ist mit einem Resolver verknüpft, welcher die Logik für den Datenabruf enthält.

Abfragen, auch \texttt{Queries} genannt, sind Anfragen des Clients an den Server. Sie beschreiben, welche Daten der Client abrufen möchte. Anders als bei REST werden alle Anfragen an eine einzige GraphQL-Schnittstelle gesendet. Von dort aus validiert der Server
anhand der Schemata, was angefragt wurde, und bearbeitet diese.
Für die Veränderung von Daten sind \texttt{Mutationen} verantwortlich. Diese Vorgänge ersetzen die \texttt{POST}-, \texttt{PUT}-, \texttt{PATCH}- und \texttt{DELETE}-Methoden, die von REST bekannt sind. Analog zu den Abfragen werden auch Mutationen in Schemata validiert.
\cite{ibm_graphql_2024}
\subsection{React}
React ist eine von Meta entwickelte JavaScript-\gls{Bibliothek} zur Entwicklung von Webanwendungen und \glspl{SPA}. Die Bibliothek basiert auf einem komponentenorientierten Ansatz, bei dem die Oberfläche in mehrere wiederverwendbare Komponenten unterteilt wird. Dafür
führte React die Syntaxerweiterung \texttt{JSX} ein, die JavaScript und eine HTML-ähnliche Syntax in einer Datei vereint. React hat sich seit seiner Einführung im Jahr 2013 zur populärsten Frontend-Bibliothek entwickelt und als maßgebliche Methode für die Entwicklung moderner,
interaktiver Benutzeroberflächen etabliert.
\subsection{Microfrontends} \label{sec:microfrontends_intro}
Analog zum Konzept der serverseitigen Microservices zerlegen Microfrontends das Frontend in mehrere semiautonome Komponenten. Die grundlegende Idee dahinter ist, ein komplexes Gesamtsystem in kleinere, überschaubare Teilbereiche zu unterteilen und diese getrennt zu bearbeiten. Dabei bringt diese Architektur sowohl Vorteile als auch Nachteile und Herausforderungen mit sich.
\textbf{Vorteile} \\
\begin{itemize}
    \item \textbf{Unabhängige Entwicklung und Bereitstellung}: Teams können einzelne Microfrontends innerhalb der Gesamtanwendung unabhängig voneinander entwickeln und bereitstellen.
    \item \textbf{Technologieunabhängigkeit}: Teams haben freie Wahl der Technologien, die ihren Anforderungen gerecht werden.
    \item \textbf{Fehlerisolation}: Bei Fehlern fällt nur das betroffene Microfrontend aus, nicht die gesamte Anwendung.
    \item \textbf{Skalierbarkeit}: Während bei Lastspitzen monolithische Anwendungen als Ganzes hochskaliert werden müssen, können bei Microfrontends gezielt einzelne Teilsysteme hochskaliert werden.
\end{itemize}
\cite[Kap.~2.2.1]{marktl2023micro}\\
\textbf{Nachteile} \\
\begin{itemize}
    \item \textbf{Kommunikation}: Kommunikation, die über die Grenzen einzelner Microfrontends hinausgeht, erfordert zusätzliche Komplexität \cite[Kap.~2.3.1]{marktl2023micro}.
    \item \textbf{Verwaltung gemeinsamer Ressourcen}: Gemeinsam genutzte Ressourcen müssen über die einzelnen Microfrontends hinweg koordiniert und geteilt werden \cite[Kap.~2.3.2]{marktl2023micro}.
    \item \textbf{Einheitlichkeit}: Da die Microfrontends unabhängig voneinander entwickelt werden, muss eine Konsistenz der Benutzeroberfläche und der Anwendungsarchitektur gewährleistet werden \cite[Kap.~1.1]{marktl2023micro}.
\end{itemize}
\subsubsection{Module Federation}
Module Federation ist ein von Zack Jackson erfundenes Architekturmuster, das 2020 mit der Veröffentlichung von Webpack 5 eingeführt wurde. Es bietet die Möglichkeit, Code externer Anwendungen zur Laufzeit zu laden und somit monolithische Anwendungen
in Microfrontends zu unterteilen. Dabei wird eine Anwendung als \textit{Host} definiert, während die restlichen als \textit{Remotes} fungieren. Während der Laufzeit der Hostanwendung kann diese bei Bedarf auf Code bzw. Komponenten der \textit{Remotes} zugreifen, die von diesen aktiv bereitgestellt werden (siehe \ref{fig:mf_vergleich_konfig}).
\begin{figure}[H]
    \centering
    \begin{minipage}{0.48\textwidth}
        \begin{lstlisting}[language=javascript, caption={Host}, label={lst:host}]
{
 import('mfe1/Cmp')

 remotes: {
  mfe1: "mfe1"
 }
}
        \end{lstlisting}
    \end{minipage}
    \hfill
    \begin{minipage}{0.48\textwidth}
        \begin{lstlisting}[language=javascript, caption={Remote}, label={lst:remote}]
{
  exposes: {

    ./Cmp: "./my.cmp.ts"

  }
}
        \end{lstlisting}
    \end{minipage}
    \caption{Webpack-Konfiguration für Host und Remote (Quelle: \cite{heise_mf})}
    \label{fig:mf_vergleich_konfig}
\end{figure}


Eine wesentliche Besonderheit von Module Federation besteht darin, dass Abhängigkeiten der einzelnen Anwendungen untereinander geteilt werden können \cite{heise_mf}, wodurch sich die Bundle-Größen erheblich verringern und die Effizienz gesteigert wird.
\begin{figure}[H]
    \centering
    \begin{minipage}{0.48\textwidth}
        \begin{lstlisting}[language=javascript, caption={Host}, label={lst:host_shared}]
{
 shared: [
  "@angular/core", "..."
 ]
}
        \end{lstlisting}
    \end{minipage}
    \hfill
    \begin{minipage}{0.48\textwidth}
        \begin{lstlisting}[language=javascript, caption={Remote}, label={lst:remote_shared}]
{
  shared: [
  "@angular/core", "..."
 ]
}
        \end{lstlisting}
    \end{minipage}
    \caption{Module Federation erlaubt das Teilen von Bibliotheken zwischen einzelnen Anwendungen (Quelle: \cite{heise_mf})}
    \label{fig:mf_vergleich_depend}
\end{figure}
