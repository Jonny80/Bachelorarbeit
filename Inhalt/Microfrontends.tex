\section{Modulare Softwarearchitektur mittels Module Federation}
Jede Großbaustelle hat ihre eigenen individuellen Herausforderungen. Da jedoch viele Softwarelösungen als Allround-Lösungen konzipiert sind,
scheitern sie oft an den spezifischen Detailanforderungen oder sind zu komplex und können so ihr Potenzial bei Weitem nicht ausschöpfen \cite[Kap.~3.1]{feth2023studie}.
In der Praxis führt dies oft zu individuellen Spezialaufträgen an die Softwarehersteller - ein Prozess, der meist mit erheblichen Zusatzkosten und langen Entwicklungszeiten
verbunden ist \cite[Kap.~4.1]{feth2023studie}.
Um dem entgegenzuwirken, bedarf es einer flexiblen und vor allem einfach erweiterbaren Struktur, in welcher individuelle Softwarelösungen als Erweiterungen implementiert werden können.
Ein möglicher Ansatzpunkt hierfür ist das Konzept der Microfrontends. Dabei werden einzelne Funktionalitäten als eigenständige \glspl{Webapplikation} entwickelt,
die sich zu einer gemeinsamen Benutzeroberfläche zusammenfügen lassen. Dabei fungiert eine Webapplikation, die sogenannte Shell, als Container, in welche die einzelnen
Microfrontends geladen werden. In Bezug auf eine Inbetriebnahmemanagementsoftware können die einzelnen Microfrontends exemplarisch eine \ac{ERP}-Software darstellen oder für
einzelne Gewerke wie \ac{MSR} oder \ac{TGA} fungieren. Wie bereits in Kapitel \ref{sec:microfrontends} erwähnt, bietet dieser Ansatz mehrere Vorteile. Somit kann gewährleistet werden,
dass bei Änderungen oder Updates nicht das Gesamtsystem beeinträchtigt wird und so ein kontinuierlicher Betrieb auf der Baustelle sichergestellt werden kann.
Des Weiteren bietet diese Architektur auch bei Lastspitzen einzelner Gewerke durch beispielsweise Prüfprozesse oder Mängelbehebungen Vorteile. So ist es möglich einzelne
Microfrontends unabhängig voneinander zu skalieren.
\subsection{Konzeptioneller Aufbau einer Microfrontend-Architektur}
Der Aufbau einer Microfrontend-Architektur zeichnet sich durch eine strikte Trennung einzelner Funktionalitäten aus. Kernfunktionalitäten wie die Authentifizierung oder das Routing werden meist
im Host abgebildet, welcher auch als Einstiegspunkt der Webapplikationen dient. Da die einzelnen Remotes als Komponenten geladen werden, können mehrere dieser auch simultan auf einer Oberfläche dargestellt werden.
Hierbei wird grundlegend zwischen einem horizontalen und einem vertikalen Ansatz unterschieden. Bei dem vertikalen Ansatz wird nur ein Microfrontend pro Seite geladen, während bei einem horizontalen Ansatz mehrere
Microfrontends simultan auf einer Oberfläche dargestellt werden können \cite[Kap.~3.1]{taibi2022micro}.
In Abbildung \ref{fig:micro_frontends} wird ein horizontaler Aufbau exemplarisch dargestellt:

dynamisch geladen.
\subsection{Routing}
Eine der zentralen Säulen einer Webapplikation ist das Routing, also das Zuordnen einer URL zu einem Zustand der Benutzeroberfläche. Hierbei ist zwischen der URL-basiertem und dem zustandsbasierten Routing zu unterscheiden.
Während bei URL-basiertem Routing die Änderung der Ansicht mit einer Anpassung der URL einhergeht, wird bei einem zustandsbasierten Routing die aktuelle Ansicht als Variable innerhalb des Clients definiert. Durch den Einsatz
von \textit{Module Federation} werden die Einzelnen \textit{Remotes} als eigenständige Komponenten herausgeführt. Dadurch bleibt die Flexibilität, wie bei einer monolithischen Webapplikation, bestehen. Es kann frei gewählt werden,
welcher Routing-Ansatz genutzt werden soll (vgl. Abbildung \ref{lst:microfrontends_routing}).
\subsection{Kommunikation zwischen Microfrontends}
In bestimmten Szenarien ist eine direkte Kommunikation zwischen den Microfrontends notwendig. Wie bereits in Kapitel \ref{sec:microfrontends} erwähnt, stellt dies eine größere Herausforderung für Microfrontends, als für monolithische Anwendungen dar.
Um dieser Hürde entgegenzuwirken, gibt es nach \cite[Kap.~4]{microservices_state_management} mehrere Lösungsansätze: \newline
\begin{itemize}
  \item \textbf{Eventgesteuerte Kommunikation}: Verwendung des \gls{Interface} \gls{eventemitter}. Dieser basiert auf dem Observer-Pattern und erlaubt über einen Publish/Subscribe-Mechanismus den Datenaustausch innerhalb des Browsers. Ein Microfrontend löst ein Event aus, auf das abonnierte Microfrontends reagieren können, um gegebenenfalls ihre Daten zu aktualisieren.
  \item \textbf{API-basierte Kommunikation}: Verlagert die Synchronisationslogik auf den Server. Die Microfrontends kommunizieren ausschließlich über API-Aufrufe mit dem Server. Datenänderungen werden zuerst mit dem Server synchronisiert, bevor andere Microfrontends diese abfragen können.
  \item \textbf{Geteilte Bibliotheken oder Services}: Nutzt geteilte Bibliotheken oder eigenständige Microfrontends für den Datenaustausch. Da Microfrontends nicht zwingend eine Benutzeroberfläche besitzen müssen, können sie auch als Logikschicht fungieren (z.~B. als \gls{middleware} für den übergeordneten Datenaustausch).
  \item \textbf{Framework-unabhängiges State-Management}: Nutzt technologieübergreifende Mechanismen wie native, browserbasierte Speicher (z.~B. \gls{local_storage}) für den Datenaustausch.
\end{itemize}
Welcher dieser Ansätze gewählt oder ob eine Kombination aus mehreren Methoden implementiert wird,
hängt letztlich stark von den spezifischen Anforderungen der jeweiligen Softwarelösung ab.
Ein Beispiel wurde im Microfrontend \textit{communication} umgesetzt, unter dem Pfad \textit{building\_management/apps/communication/src/App.tsx} auf dem beigelegten USB-Stick \cite{eigener_code_usb}.

\subsection{Organisation der Applikationen} \label{monorepo_orga}
Um die einzelnen Codebasen der Microfrontends übersichtlich zu verwalten und eine durchgehende Konsistenz hinsichtlich einheitlicher \ac{UI}-Elemente zu gewährleisten, wurde das Projekt als Monorepo aufgesetzt.
Ein Monorepo ist eine Strategie, bei welcher mehrere Codebasen in einem \Gls{vcs}, wie z.B. Git, verwaltet werden. Dabei können auch Hilfsmittel wie UI-Bibliotheken, Konfigurationsdateien oder Dokumentationen zentral
verwaltet werden \cite{wieruch2024monorepo}. Hierbei können die einzelnen Remotes und der Host die UI-Bibliothek direkt importieren \cite{turborepo_internal_packages}. Die Namenskonvention des Importpfades ist dabei,
spezifisch für das Node-Ökosystem, folgend definiert (siehe Listing \ref{lst:microfrontends_ui}):
\textit{workspace} sagt dem Paketmanager, dass es sich um ein lokal verlinktes Paket handelt, während das \textit{@} Präfix einen sogenannten \textit{Scope} definiert \cite{npm_doku}. Dies gruppiert zusammenhängende
Paketnamen und schützt vor Namenskonflikten, um ein Herunterladen gleichnamiger Pakete aus öffentlichen Quellen zu verhindern.
\begin{lstlisting}[language=Javascript, caption={Einbindung der UI-Bibliothek}, label={lst:microfrontends_ui}]
import { Button } from "@workspace/ui/components/button";
\end{lstlisting}

In der folgenden Abbildung wird eine mögliche Ordnerstruktur am Beispiel des verwendeten Monorepos (\gls{turborepo}) verdeutlicht:
\begin{figure}[H]
  \centering
  \includegraphics[width=1\textwidth]{Monorepo_Example.png}
  \caption{Exemplarischer Aufbau eines Monorepos}
  \label{fig:monorepo}
\end{figure}
Zusätzlich ermöglicht das Verwalten mehrerer Codebasen in einem zentralen Repository App-übergreifende Startscripts zu schreiben, um mehrere Anwendungen, wie beispielsweise die Host-App mit zwei Remote-Apps,
parallel auszuführen. Auch die Paketverwaltung wird optimiert - da \gls{npm}-Pakete zentral installiert werden, können mehrere Apps darauf zugreifen, was einer Mehrfachinstallation entgegenwirkt.

\subsection{Konfiguration von Module Federation in React}
In dem folgenden Kapitel wird die Herangehensweise einer Konfiguration der Microfrontends unter Verwendung von React und Vite beschrieben. Das Projekt selber wird in einem Monorepo (hier turborepo) realisiert.
Hierbei notwendig ist die Entwicklungsumgebung \textit{NodeJS} (Version >= 22.23.1, siehe \url{https://nodejs.org/}) und der dabei enthaltene Paketmanager \textit{npm}. Dieser wird genutzt um Pakete wie Turborepo oder Vite zu verwalten.
Die Implementierung wird in mehrere Schritte unterteilt. Zuerst wird kurz gezeigt wie das Turborepo aufgesetzt wird, um eine organisatorische Grundlage für die folgenden Schritte zu schaffen. Anschließend werden
zwei React Projekte mit Vite als Build-Tool erstellt. Einer als Host und einer als Remote App. Dem folgend wird gezeigt wie mithilfe des Plugins \textit{@module-federation/vite} die Microfrontends konfiguriert werden.
Der letzte Schritt zeigt exemplarisch die Einbindung der Remote-App in der Host-App.
\subsubsection{Aufsetzen von Turborepo}
Das Monorepo Turborepo wird mithilfe des folgenden Befehls aufgesetzt:
\begin{lstlisting}[language=Shell, caption={Aufsetzen von Turborepo}, label={lst:microfrontends_turborepo}]
npx create-turbo@latest -e with-vite
\end{lstlisting}
Wobei der Zusatzbefehl \textit{-e with-vite} das Monorepo direkt mit einer Vite-Umgebung initialisiert. Die Orderstruktur wird in \ref{monorepo_orga} verdeutlicht (vgl. Abbildung \ref{fig:monorepo}).
Die \textit{package.json} im Grundverzeichnises des Repositories steuert hierbei unter anderem die Ausführung der einzelnen Apps, wie beispielsweise der Start der Host App und der Remote App, durch den folgenden Befehl:
\begin{lstlisting}[language=Shell, caption={Starten der Apps}, label={lst:microfrontends_apps_start}]
turbo run dev --filter=< Name > --filter=< Name > ... --filter=< Name >
\end{lstlisting}
Der Ausdruck < Name > steht hier stellvertretend als der Name des Ordners der jeweiligen App im \textit{/app} Unterverzeichnis im Monorepo.
\subsubsection{Konfiguration der Microfrontends}
Die Konfiguration der Microfrontends wird hier unter der Verwendung des offiziellen Vite-Plugin \textit{@module-federation/vite} realisiert. Dieses Plugin vereinfacht die Konfiguration im Vergleich zu Webpack erheblich.
Dabei stellt \textit{Module Federation} für viele gängigen Webtechnologien Plugins zur Verfügung (vgl. \cite{mf_integrations_guide}).
In Vite wird das Plugin mit dem folgenden Befehl in der jeweiligen App installiert:
\begin{lstlisting}[language=Shell, caption={Installieren des Vite-Plugins}, label={lst:microfrontends_plugin_install}]
npm add @module-federation/vite --save
\end{lstlisting}
